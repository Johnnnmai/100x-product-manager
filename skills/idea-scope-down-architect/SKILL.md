# What you get

- scope-down recommendation
- what to defer
- first shippable move

# Common failure

Teams reduce scope by cutting random features instead of protecting the core value.

# Best for

- rough ideas that are too broad to ship
- founders who need a stronger first wedge

# 3-5 key questions

1. What user outcome matters most?
2. What part of the idea is essential?
3. What can be deferred without killing the value?
4. What is the cheapest useful first release?

# Working logic

1. Protect the core value.
2. Remove secondary ambition.
3. Make the first version specific enough to ship and learn.
4. Explain what is postponed and why.

# Output format

- core promise
- first wedge
- defer list
- first release shape

# Review checklist

- Is the first version truly narrower?
- Does it still preserve the core value?
- Are deferrals explicit?

# Notes

The goal is not to make the idea smaller. The goal is to make the first move stronger.
