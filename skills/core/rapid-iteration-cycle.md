# Rapid Iteration Cycle / 快速迭代循环

[English](#english) | [中文](#中文)

---

## English

### When to Use This Skill

Use this when you need to validate an idea, feature, or hypothesis **fast** (within 2 weeks).

### Core Question

**"How do we validate an idea in 2 weeks?"**

### The Framework

#### Phase 1: Define (Day 1)

**What are we testing?**
- State the hypothesis clearly
- "If [action], then [metric] will [change] because [reason]"

**What's the simplest test?**
- Minimum Viable Test (MVT), not MVP
- What's the smallest thing that tests this hypothesis?

**What's our success metric?**
- One primary metric
- Define threshold: what level of improvement proves the hypothesis?

#### Phase 2: Build (Days 2-5)

**Speed Principles:**
- Use existing components
- Fake it if needed (no real backend)
- 80/20: 20% effort for 80% learning

**Deliverable:**
- Not a perfect product
- A testable hypothesis in front of real users

#### Phase 3: Test (Days 6-12)

**Run the experiment:**
- Get it in front of users ASAP
- Minimum 100 users or 7 days of data
- Don't peek! (statistical validity)

**Gather evidence:**
- Quantitative: Metrics, conversion, usage
- Qualitative: Feedback, interviews, observations

#### Phase 4: Decide (Days 13-14)

**Analyze:**
- Did we hit the success threshold?
- What did we learn?
- What surprised us?

**Decide:**
| Result | Action |
|--------|--------|
| ✅ Validated | Scale it |
| 🟡 Partial | Iterate fast |
| ❌ Invalidated | Pivot fast |

### The 2-Week Sprint Template

| Day | Focus | Output |
|-----|-------|--------|
| 1 | Define hypothesis | Clear testable hypothesis |
| 2-3 | Design MVT | Test design + success criteria |
| 4-5 | Build fast | Working test (even if rough) |
| 6-12 | Run test | Data + feedback |
| 13-14 | Analyze + decide | Go/No-go recommendation |

### Key Principle

**Move Fast, Iterate Faster:**
- A bad test is better than no test
- Speed reveals what matters
- Iteration compounds learning

## 中文

### 何时使用此技能

当需要**快速**验证一个想法、功能或假设（2周内）时使用。

### 核心问题

**"我们如何用2周验证一个想法？"**

### 框架

#### 第一阶段：定义（第1天）

**我们在测试什么？**
- 清晰陈述假设
- "如果[行动]，那么[指标]将[变化]因为[原因]"

**最简单的测试是什么？**
- 最小可行测试 (MVT)，不是 MVP
- 什么是最小能验证这个假设的东西？

**我们的成功指标是什么？**
- 一个主要指标
- 定义阈值：什么程度的改进证明假设成立？

#### 第二阶段：构建（第2-5天）

**速度原则：**
- 使用现有组件
- 如需要可以伪造（不需要真正的后端）
- 80/20: 20%努力获得80%学习

**交付物：**
- 不是完美的产品
- 在真实用户面前可测试的假设

#### 第三阶段：测试（第6-12天）

**运行实验：**
- 尽快让用户看到
- 最少100用户或7天数据
- 不要偷看！（统计有效性）

**收集证据：**
- 定量：指标、转化、使用
- 定性：反馈、访谈、观察

#### 第四阶段：决定（第13-14天）

**分析：**
- 我们达到成功阈值了吗？
- 我们学到了什么？
- 什么让我们惊讶？

**决定：**

| 结果 | 行动 |
|------|------|
| ✅ 验证通过 | 规模化 |
| 🟡 部分验证 | 快速迭代 |
| ❌ 验证失败 | 快速转向 |

### 2周冲刺模板

| 天数 | 重点 | 输出 |
|------|------|------|
| 1 | 定义假设 | 清晰可测试的假设 |
| 2-3 | 设计 MVT | 测试设计 + 成功标准 |
| 4-5 | 快速构建 | 可工作的测试（即使粗糙） |
| 6-12 | 运行测试 | 数据 + 反馈 |
| 13-14 | 分析 + 决定 | 推进/不推进建议 |

### 关键原则

**快速行动，更快迭代：**
- 糟糕的测试优于没有测试
- 速度揭示什么重要
- 迭代累积学习
