# 100X PM Core Sell Points

This document defines what 100X PM should actually sell, what should stay subordinate, and what needs proof.

## One-Line Positioning

**100X PM helps PMs and founders turn messy ideas, existing products, and fragmented strategy context into sharper direction, stronger business choices, and agent-ready execution.**

Canonical line:

**Scrape the market. Simulate the users. Compile the strategy.**

## What Users Think They Are Buying

At first glance, many users will think they are buying one of these:

- an AI idea validator
- a PRD generator
- a PM prompt library
- a faster way to write product docs

Those are not the real sell points.

## What They Are Actually Buying

### 1. Faster convergence from ambiguity to decision

The first real sell point is not writing speed. It is **decision compression**.

100X PM should help users move from vague input to:

- a clear wedge
- explicit trade-offs
- a business model choice
- a first channel choice
- a next execution path

### 2. Simulate before you build

The second real sell point is **pre-build strategy validation**.

The differentiated chain is:

- `Scrapling` for public signals
- persona / market-twin construction
- `OASIS` for reaction and scenario simulation
- strategy synthesis into PM decisions

The promise is not certainty.
The promise is **lower-cost uncertainty reduction before engineering effort is committed**.

### 3. Compile strategy into execution

The third real sell point is that the output does not stop at analysis.

100X PM should compile direction into:

- PRDs
- roadmaps
- experiments
- metrics
- delegation-ready tasks for humans or agents

## Three User-Facing Entry Points

### Product audit

What users buy:

- hidden pain-point discovery
- sharper hypotheses
- better product bets from a real website or app

### New idea shaping

What users buy:

- wedge selection
- business model clarity
- first channel and first move

### Long-range vision

What users buy:

- 3-5 year direction from messy context
- strategic narrative
- phased roadmap

## Table Stakes vs Differentiation

### Table stakes

Useful, but not enough to anchor the product story:

- AI writes PRDs faster
- templates reduce blank-page effort
- commands create more structured output

### Differentiation

Claims worth leading with:

- uses external signals instead of only internal opinion
- uses simulation before build, not only after launch
- separates strategic output from execution output
- compiles decisions into agent-ready execution

## What We Should Not Claim

- do not imply perfect prediction
- do not imply OASIS removes the need for product judgment
- do not imply simulation is a substitute for real users
- do not sell "AI writes faster" as the core moat

## Proof We Need

| Sell point | Proof type |
| --- | --- |
| Faster convergence | fewer clarification turns, clearer first draft, better next-step clarity |
| Simulate before build | better evidence quality, fewer hidden assumptions, stronger risk articulation |
| Compile into execution | stronger PRD/roadmap readiness, fewer missing decisions, lower manual restructuring |

## Current Messaging Hierarchy

Lead with:

1. `product audit` as the hero workflow
2. `new idea` as the founder growth path
3. `long-range vision` as the premium strategic mode

Support with:

- Lighty as the routing guide
- OASIS as the simulation layer
- Paperclip / AI OS as the execution layer

## Summary

If the repo sounds like "a better prompt library," the positioning is slipping.

If it sounds like "a market-twin-powered strategy compiler that turns messy inputs into decisions and execution," the positioning is working.
